🚀 上线 + 访问门禁

OSS/CDN 部署 · 飞书登录门禁 · 变更通知（含踩过的坑）
文档写好后，怎么让团队访问、怎么只让组织内部人能看、怎么在更新后自动通知。
本文是这套方法落地时踩过坑后的实操总结。

一、上线：两种姿势

文档站是纯静态 HTML（docs/*.html + docs/assets/），没有构建步骤，部署很轻。

A. OSS + CDN（推荐：对外 / 多人 / 全球加速）

文档扔对象存储（OSS），CDN 加速 + 绑域名。

export ALIYUN_ACCESS_KEY_ID=xxx ALIYUN_ACCESS_KEY_SECRET=xxx   # 需 OSS 读写 + CDN 刷新权限
export OSS_BUCKET=your-doc-bucket
export CDN_DOMAIN=docs.example.com      # 可选；留空只传 OSS、不刷 CDN
bash scripts/deploy-oss-cdn.sh

脚本做的事：

  • docs/assets/（mermaid 等）→ 长缓存 max-age=31536000
  • 其余 *.html → no-cache，改了就即时生效（关键，见「坑 ①」）
  • CDN_DOMAIN 非空时调 refresh-cdn.py 刷阿里云 CDN
CDN 是哪家？ 阿里云 CDN 用 scripts/refresh-cdn.py（脚本自动调）；七牛云 / 其他 CDN 上传 OSS 后去各自控制台刷缓存（脚本会提示）。

B. ECS / VPS nginx 直连 + Let's Encrypt（推荐：自主可控 / 和后端同机）

DNS A 记录指到服务器 → nginx 托管 docs/ 静态 + certbot 签 HTTPS。

server {
  server_name docs.example.com;
  root /var/www/your-docs;        # 把 docs/ 同步到这里
  index index.html;
  location / { try_files $uri $uri/ /index.html; }
  # certbot --nginx -d docs.example.com 自动补 443 + 80→443 跳转
}

和后端同机的最大好处：门禁可以「同源」，省掉所有跨域 CORS 麻烦（见下）。

二、访问门禁：飞书登录（只让组织内部人看）

静态站本身不能鉴权。做法是一段 JS 门禁（docs/assets/auth-guard.js）在页面渲染前拦截：

①无 session → 跳 后端 /auth/feishu/login → 飞书授权页
②飞书回调后端 /callback → 后端校验 tenant_key（组织标识）→ 建 session → 跳回文档站带 session
③门禁 JS 存 session → 调 后端 /verify 校验 → 通过才显示页面；失败回 ①

需要三样东西

  1. 一个后端服务，提供三个接口：
    • GET /auth/feishu/login?redirect=<回跳地址> → 302 到飞书授权页
    • GET /auth/feishu/callback?code&state → 换 token、校验组织、建 session、跳回
    • GET /auth/feishu/verify（带 Authorization: Bearer <session>）→ 200/401
    • （这套接口和具体后端技术栈绑定，本规范不含后端代码；逻辑见本文「后端接口要点」）
  2. 飞书开放平台配置：
    • 拿 app_id / app_secret
    • 「安全设置 → 重定向 URL」白名单加 {后端}/auth/feishu/callback
    • 组织 tenant_key（登录后校验用户属不属于本组织）
  3. 门禁脚本 docs/assets/auth-guard.js：把里面的后端 base url 改成你的。

后端接口要点（伪代码，照搬到任意语言）

login:    state=随机存Redis(5min); redirect_uri = scheme://本次请求Host + /auth/feishu/callback
          → 302 https://open.feishu.cn/open-apis/authen/v1/authorize?app_id&redirect_uri&state
callback: 校验state; code 换 app_access_token → user_access_token → user_info
          if user_info.tenant_key != 配置的组织tenant_key: 403
          sessionId=随机; Redis存 session(7d); 302 回 redirect 带 ?session=sessionId
verify:   读 Authorization: Bearer; Redis 查 session; 命中 200 否则 401

⚠️ 踩过的坑（务必看）

坑说明对策
① CDN 缓存旧门禁门禁 JS / index.html 被 CDN 缓存，改了不生效这俩必须 no-cache（部署脚本已做）；CDN 该刷还得刷
② tenant_key ≠ 企业编号企业编号长 FExxxxxxx，不是 tenant_keytenant_key 是 OAuth user_info 返回的 16 位 hex；先跑一次登录从日志抓真值
③ callback 域名写死一套后端服务多个前端域名时，回调会跳错域名login 用 X-Forwarded-Host 动态生成 callback，跟随入口域名；仅信任你的主域防注入
④ 跨子域 CORS前端 docs.x.com 调后端 api.x.com/verify 被 CORS 拦后端放行 https://*.x.com；或前端后端同机同源（姿势 B），彻底免 CORS
⑤ 协议不一致http 文档站调 https 后端，preflight 跟随重定向失败门禁 JS 里 apiBase 固定 https

三、变更通知（可选）：文档更新发飞书

CI 里文档 push 后跑 scripts/feishu-doc-change-notify.mjs，往飞书群发一张卡片（仓库 / 提交 / 改了哪些文件 / 一键查看链接）。

# 二选一：群机器人 webhook
export FEISHU_WEBHOOK_URL=https://open.feishu.cn/open-apis/bot/v2/hook/xxx
export FEISHU_WEBHOOK_SECRET=xxx                 # 可选（群机器人开了签名校验才需要）
# 或：应用机器人
export FEISHU_APP_ID=cli_xxx FEISHU_APP_SECRET=xxx FEISHU_CHAT_ID=oc_xxx

export DOC_CHANGE_SUMMARY="$(git log -1 --pretty=%s)"
export DOC_CHANGE_FILES="$(git diff --name-only HEAD~1 HEAD)"
export DOC_CHANGE_URL="https://github.com/org/repo/commit/$(git rev-parse HEAD)"
node scripts/feishu-doc-change-notify.mjs

干跑预览：FEISHU_NOTIFY_DRY_RUN=1 node scripts/feishu-doc-change-notify.mjs（只打印卡片 JSON 不发送）。

四、最小上线 checklist

  1. 发布：export OSS_BUCKET + ALIYUN key → bash scripts/deploy-oss-cdn.sh
  2. （可选）门禁：起后端三接口 + 飞书 app 配置 + 改 auth-guard.js 的 apiBase
  3. （可选）通知：CI 加一步 feishu-doc-change-notify.mjs
  4. 团队只记一个地址：文档站的 index.html
没有门禁需求就跳过第 2 步——文档站照样能用，只是公开可访问。
